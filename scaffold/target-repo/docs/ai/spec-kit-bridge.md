## Cross-Agent Review Safety
- User-level standing approval for invoked cross-agent reviews covers only the relevant review context: active Spec Kit artifacts, changed files, implementation notes, and validation evidence for the feature being reviewed.
- Do not share secrets, credentials, tokens, raw environment values, unrelated private files, or unrelated repo content with a peer agent.
- Minimize context before sending it to a peer agent; prefer paths, summaries, focused excerpts, and reviewed diffs when full file content is not needed.
- Consent to share review context is separate from permission to run peer CLI smoke tests, install tools, call networked services, or request sandbox/network escalation.

## Ownership Boundary
- Treat official Spec Kit files as Spec Kit-owned: `.specify/`, `specs/`, and official generated prompt, agent, script, template, or skill files.
- Treat the managed Spec Kit bridge summary inside [AGENTS.md](../../AGENTS.md) as enhancer-owned and safe for future bridge refreshes.
- Treat this file and any installed bridge skills as enhancer-managed guidance that may be regenerated by later enhancer refreshes.
- Treat enhancer-owned bridge skills as `.codex/skills/` files only. If `.agents/skills/` exists, treat it as an external compatibility surface; do not mirror, migrate, or overwrite files there.
- Treat detected generic command directories, presets, extensions, script folders, and git extension files as official Spec Kit-owned surfaces.

## Diagnostics
- `codex-enhancer spec-doctor <repo>` reads local Spec Kit state without running the Spec Kit CLI.
- `codex-enhancer spec-doctor <repo> --check-spec-kit-cli` may run local read-only `specify version`, `specify version --features --json`, and `specify integration list` diagnostics.
- Normal enhancer install, upgrade, refresh, `spec-report`, and `spec-sync` flows must not run `specify`.

## Official Operations
- Use official Spec Kit commands for integration `list`, `install`, `uninstall`, `switch`, `use`, and `upgrade`.
- Use official Spec Kit commands for preset and extension add, remove, enable, disable, update, catalog, priority, and configuration workflows.
- Do not use enhancer refresh or bridge-management flows to add, remove, upgrade, or force-delete official Spec Kit-managed files.
- If the repo is not a Git worktree or the git extension is absent, official Spec Kit feature commands may need `SPECIFY_FEATURE`; add or remove the git extension through Spec Kit, not the enhancer.

## Working Rule
- When a feature already has Spec Kit artifacts, read the active spec and plan before implementing code.
- Keep implementation, validation, and review notes aligned with the current feature artifacts instead of only the latest prompt.
- Do not use enhancer flows to rewrite official Spec Kit templates or feature history.
